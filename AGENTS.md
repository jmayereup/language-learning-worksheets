## 1. What this project is

A two-mode web app for **Teacher Jake's interactive language learning worksheets**:

1. **Hosted library** (the main React SPA) — browse, search, and play worksheets
2. **Embeddable web components** — each lesson is also a self-contained custom element that can be pasted into WordPress / Ghost / Notion

Lesson content is generated by Gemini (via pre-prompted gems) and stored in PocketBase. The compiled HTML (`htmlCompiled`) is what gets embedded externally; the React app always re-derives the equivalent at runtime from the raw fields.

## 4. Data model

### 4.1 PocketBase collection: `worksheets`

The schema is auto-generated into `pocketbase-types.ts` (do not hand-edit — re-run `pocketbase-typegen` if the schema changes). The key fields are:

| Field | Type | Notes |
| --- | --- | --- |
| `title` | text | Lesson title |
| `language` | select (enum) | `WorksheetsLanguageOptions` — e.g. `English`, `Spanish`, … |
| `level` | select (enum) | `WorksheetsLevelOptions` |
| `tags` | multi-select | `WorksheetsTagsOptions` |
| `lessonType` | select (enum) | **Drives the WC bundle + the React dispatch in `LessonView.tsx`** — see §5 |
| `content` | json | The actual lesson data. JSON or HTML custom element (see §6) |
| `html` | text | Free-form HTML shown above/below the component (TinyMCE-edited) |
| `htmlCompiled` | text | Pre-rendered embed artifact. Auto-derived on save; not consumed by the React app |
| `teacherCode` | text | Auto-applied to WC as `code` attr (defaults to `"6767"`) |
| `customConfig` | json | e.g. `{ "testMode": true }` — see `getAutoAttrs()` below |
| `image` / `audioFile` | file | Uploaded assets — URLs via `getFileUrl()` |
| `videoUrl` | url | YouTube link |
| `isVideoLesson` | bool | UI badge + filter |
| `seo` | text | Lesson-list snippet (auto-synced from `content.seo_intro` when empty) |
| `creatorId` | relation → users | Set server-side from auth in `createLesson` |

Enums: see `WorksheetsLessonTypeOptions` (`pocketbase-types.ts:237`).

### 4.2 The `content` field — format rules (the §6 system)

`content` can be one of three formats. This is the **single most important concept** in the codebase. All entry points go through `utils/contentFormat.ts`.

| First non-whitespace char | Format | Storage | What it becomes at compile time |
| --- | --- | --- | --- |
| `{` or `[` | `'json'` | object (JSON-parsed) | wrapped in `<tj-*>` with `<script type="application/json">` |
| `<` | `'html'` | string (the raw element) | used **verbatim** as the WC element; auto-attrs appended (deduped) |
| anything else | `'invalid'` | rejected at editor level | — |

`utils/contentFormat.ts` exports:

- `detectContentFormat(raw)` — first-char check (fast)
- `parseContent(raw)` — full validation. For HTML: enforces single root element, tag contains a hyphen (custom-element spec), matching `</tag>` close. Returns a `DetectedContent` discriminated union.
- `isValidCustomElementTagName(tag)` — regex `/^[a-z][a-z0-9]*-[a-z0-9-]+$/`
- `getFormatLabel(format)` — `'JSON' | 'HTML Custom Element' | 'Invalid'`

### 4.3 `htmlCompiled` — the embed artifact

`htmlCompiled` is a **pre-rendered, self-contained HTML string** that can be pasted into a third-party site. It's:

- A `<link rel="stylesheet">` to the CDN CSS
- A `<article class="tj-worksheet-compiled">` wrapper with the title + SEO header
- The interactive element (lesson-type's `<tj-*>` OR the user-authored custom element)
- A `<script type="module" defer>` tag loading the WC bundle from the CDN
- The user's `html` instructions either side of a `<lesson-component>` / `<web-component>` placeholder, or appended at the end if no placeholder is present

The React app does NOT consume `htmlCompiled` at runtime — it always re-derives the equivalent by calling `compileLessonHtml()` from the raw `html` + `content` + `lessonType`. `htmlCompiled` exists so external embeds can be a single copy-paste.

`htmlCompiled` is computed in two places:

1. **Client (preferred):** `components/Admin/LessonEditor.tsx:228` — computed from form state and sent in the same `FormData` request
2. **Server (fallback):** `services/pocketbase.ts` `compileHtmlForRecord()` — re-runs after create/update if the caller didn't supply a pre-computed value

## 5. Lesson type → web component mapping

`utils/componentMapper.ts` is the source of truth. Each lesson type maps to a `{ tag, script }` pair:

| Lesson type | Tag | Script CDN |
| --- | --- | --- |
| `lbl-reader` | `lbl-reader` | `tj-reader.js` |
| `grammar-hearts` | `tj-grammar-hearts` | `tj-grammar-hearts.js` |
| `information-gap` | `tj-info-gap` | `tj-info-gap.js` |
| `listening` | `tj-listening` | `tj-listening.js` |
| `speed-review` | `tj-speed-review` | `tj-speed-review.js` |
| `chapter-book` | `tj-chapter-book` | `tj-chapter-book.js` |
| `pronunciation` | `tj-pronunciation` | `tj-pronunciation.js` |
| `quiz-element` | `tj-quiz-element` | `tj-quiz-element.js` |
| `worksheet` / `focused-reading` / `word-blaster` | *(none — React-only or `tj-pocketbase-worksheet`)* | falls back to inline React `WorksheetView` or the bundled `<tj-pocketbase-worksheet>` element |

The React app injects the CDN script dynamically (no bundling):

- `components/Lesson/LessonView.tsx:91` — `<script id="script-view-${type}">` (idempotent)
- `components/Lesson/WebComponentPreview.tsx:57` — `<script id="script-preview-${type}">`

**The script is always determined by the lesson type, even when the user provides a custom element.** The user's custom element gets the `code`, `lang-original`, `audio-listening`, `test-mode` auto-attrs (deduped against user-provided ones) but uses the lesson-type's WC bundle.

`getAutoAttrs()` lives in `utils/htmlCompiler.ts` (and is duplicated in `buildPreviewElementHtml()` — keep them in sync):

- `code` ← `teacherCode` (string)
- `lang-original` ← `language` (only `lbl-reader`)
- `lang-translation` ← hardcoded `"Thai"` (only `lbl-reader`) — TODO: this is stale, should be `translationLanguage` from the lesson
- `story-title` ← `title` (only `lbl-reader`)
- `audio-listening` ← `audioFileUrl` (only `listening`)
- `test-mode` ← boolean from `customConfig.testMode`

## 6. The compile pipeline

`utils/htmlCompiler.ts` is the central nervous system.

### 6.1 `compileLessonHtml(lesson, rawHtml) → string`

Steps:

1. Look up `componentConfig` from `getComponentConfig(lesson.lessonType)`
2. Compute `autoAttrs` per §5
3. **Branch on content format** (via `parseContent`):
   - `'html'` → emit the user's element verbatim, append missing `autoAttrs`, then add the WC `<script src>` from `componentConfig`
   - `'markdown'` (legacy string that doesn't start with `<`) → wrap in `<tj-*>` with `<script type="text/markdown">`
   - `'json'` → wrap in `<tj-*>` with `<script type="application/json">`
4. If no `componentConfig` (i.e. lesson type is `worksheet` or unrecognized) → fall back to `<tj-pocketbase-worksheet>` with a `<script type="application/json">` containing the full record
5. Inject the element into `rawHtml` at the `<lesson-component>` / `<web-component>` placeholder if present, else append
6. Wrap in `<article class="tj-worksheet-compiled">` with header + stylesheet link

### 6.2 `buildPreviewElementHtml(lesson) → { elementHtml, contentMode }`

Same branching, but returns just the element (no script tag, no article wrapper). Used by `LessonView` and `WebComponentPreview` for the live in-app preview.

### 6.3 Round-trip semantics

The editor stores HTML content as a **string** (so PocketBase keeps it as a JSON-encoded string in the `content` field). The string is preserved on the round-trip because `JSON.stringify(string)` and the corresponding `JSON.parse` are inverses. The server-side `parseContent` in `services/pocketbase.ts:compileHtmlForRecord` detects HTML vs JSON vs legacy markdown and normalizes the value before passing it to `compileLessonHtml`.

## 7. Frontend architecture

### 7.1 View routing

There is **no React Router**. Routing is driven by URL query params (see `App.tsx:62`):

- `?view=admin` → Admin dashboard
- `?view=create` → Public "Create standalone HTML" editor (no DB writes)
- `?edit=<id>` → open admin editor for that record
- `?lesson=<id>` → show that lesson (requires login)
- `?language=`, `?level=`, `?category=` → filter state, kept in sync via `updateURL`

State is held in `App.tsx`; back/forward (`popstate`) re-syncs from the URL.

### 7.2 Data fetching

TanStack Query. Query keys are centralized in `hooks/useLessons.ts:lessonKeys` and follow the pattern `['lessons', 'list', { language, level }]` / `['lessons', 'detail', id]`.

Mutations (`useCreateLesson`, `useUpdateLesson`, `useDeleteLesson`) invalidate the appropriate keys on success.

Default stale time: 5 min, gc time: 30 min, retry: 1, no refetch on window focus.

### 7.3 Lesson dispatch

`components/Lesson/LessonView.tsx:48` picks the React view based on `effectiveLessonType`:

```ts
const effectiveLessonType = lesson.lessonType || (
  isBlaster ? 'word-blaster' :
  isFocused ? 'focused-reading' :
  isChapter ? 'chapter-book' :
  isInfoGap ? 'information-gap' :
  isStandard ? 'standard' :
  'worksheet'
);
```

- `'lbl-reader' | 'grammar-hearts' | 'listening' | 'speed-review' | 'pronunciation' | 'quiz-element'` → render the external WC inline (`dangerouslySetInnerHTML` with the bundle's `<tj-*>` element) and inject its `<script>` once
- `'information-gap' | 'focused-reading' | 'word-blaster' | 'chapter-book' | 'worksheet'` (and fallback `'standard'`) → render the corresponding React view in a `React.Suspense`

The `html` field is split at any `<lesson-component>` / `<web-component>` placeholder, and the interactive content is rendered between the parts (so users can write instructions that wrap around the component).

### 7.4 Custom-element mounting in preview

`WebComponentPreview.tsx` and `LessonView.tsx` both manually inject the WC `<script>` and use a stable script id to avoid double-loading. They also apply the `code` attribute to all custom elements (with a hyphen in the tag name) on every render of the user-authored `html` content (see `LessonView.tsx:105`).

### 7.5 The two custom-element entry points

- `pocketbase-worksheet.tsx` → defines `<tj-pocketbase-worksheet>`. Reads its child `<script type="application/json">`, mounts React `<LessonView>` in its Shadow DOM
- `chapter-book.tsx` → defines `<tj-chapter-book>`, same pattern

Both are built as a **separate** Vite library bundle (see `vite.config.ts:28`) — `npm run build:wc` produces `dist/wc/tj-*.es.js`. These are uploaded to the CDN and loaded by external embeds.

## 8. The Admin editor UX (the most-edited surface)

`components/Admin/LessonEditor.tsx` is large but the structure is:

1. **Form fields** (title, language, level, tags, video, teacher code, test mode toggle, image, audio, video-lesson flag)
2. **SEO description** textarea (auto-synced from `content.seo_intro` when empty, JSON only)
3. **HTML content** — `VisualHTMLEditor` (TinyMCE wrapper). The user can insert `<lesson-component>` / `<web-component>` as a placeholder for the interactive element
4. **Worksheet JSON Content** — the dark-themed textarea. Toolbar:
   - **Status badge** (live) — `JSON ✓` / `HTML <tag-name>` / `Invalid: <reason>`
   - **Check Format** button — runs `parseContent` and shows a green/red banner
   - **Visual Editor** button (JSON only) — opens the `JSONKeyValueEditor` modal
   - **Paste** button — reads clipboard; syncs `title` + `seo_intro` from JSON, shows detected tag name in feedback if HTML
   - **Get JSON from Gemini** — link to a Gemini gem, URL varies by lesson type
5. **Action buttons** — Cancel / Preview / Copy Embed / Save as HTML / Create|Update Online Worksheet. All except Cancel are **disabled** when content is invalid

The `LessonEditor` is also used by the public "Create" tab (`isPublicCreator={true}`) with the same UI minus the online save button — that path always goes through "Save as HTML" or "Copy Embed".

### 8.1 Where JSON parsing lives (now centralized)

All 6 historical `JSON.parse(jsonContent)` call sites in `LessonEditor.tsx` were refactored to use `parseContent` and emit a single error message. The pre-compile flow:

```ts
// buildLessonForCompile()
const detected = parseContent(jsonContent);
if (detected.format === 'invalid') throw new Error(detected.error);
const parsedContent = detected.format === 'json'
  ? applyTitleSync(detected.value, title)   // sync form title → content.title
  : jsonContent.trim();                     // HTML: pass raw string through
```

The `quiz-element` raw-string special case was removed — the new rules apply uniformly.

## 10. Deployment

This repo does **not** deploy itself. The flow is:

1. **PocketBase hooks** (`pb_hooks/*.js`) — installed on the PB server. After any create/update/delete on the `worksheets` collection, they POST to a Cloudflare Pages deploy hook
2. **Cloudflare Pages** — rebuilds the static site from the latest `dist/` (you commit dist? No — see "Manual deploy" below)
3. **Express server** (`server.js`) — wraps the built React app with dynamic OG tags for shared lesson links, and exposes `/api/rebuild` (auth via `_superusers` token) which forwards to the same Cloudflare deploy hook

> ⚠️ Per `.agents/rules/rule_deploy.md`: **never deploy on your own** — ask the user.

The PB JSVM hooks are **not** compiled; they're loaded as JS. They use `require()` (CommonJS) to share `cloudflare-rebuild.js`. The rules in `.agents/skills/pocketbase-best-practices/references/server-side-extending.md` apply — most importantly, the JSVM scope rule: each handler is serialized and executed in isolation, so module-scope state from a `require()`'d file is fine, but capturing variables outside the handler in the hook file itself does **not** work.

## 11. External embed (third-party sites)

Anyone can copy a worksheet and paste it elsewhere:

1. Admin opens a lesson, clicks **Copy Embed** or **Save as HTML**
2. The clipboard receives `htmlCompiled` (a self-contained HTML snippet with `<link>`, `<article>`, the `<tj-*>` element, and a `<script src>` to the CDN bundle)
3. The host page includes the CSS link, the snippet HTML, and the script tag. The custom element registers itself when the script loads
4. For `<tj-pocketbase-worksheet>` and `<tj-chapter-book>` the element reads its child `<script type="application/json">` payload and mounts the React view in its Shadow DOM — full styling isolation, no Tailwind dependency

CSS: hosted at `https://worksheets.teacherjake.com/wc/language-learning-worksheets.css`

## 12. Adding a new lesson type — checklist

If you ever add a new lesson type end-to-end:

1. **Schema** — add the value to the `worksheets.lessonType` enum in PocketBase; re-run `pocketbase-typegen` to refresh `pocketbase-types.ts`
2. **TypeScript type** — add the new content shape to `types.ts:LessonContent` union
3. **Mapping** — add a case in `utils/componentMapper.ts` with `{ tag, script }`
4. **External bundle** — create `dist/wc/tj-<name>.es.js` (either as a new Vite lib entry in `vite.config.ts` or as a separately hosted script)
5. **React view** — either:
   - Create a new `components/Lesson/<Name>View.tsx` and add it to the dispatch table in `LessonView.tsx`
   - Or, if the WC bundle handles it entirely, add the type to the external-WC list at `LessonView.tsx:220`
6. **Gemini gem** — if you want a "Get JSON from Gemini" link for the type, add the URL in `LessonEditor.tsx:892-903`
7. **Auto-attrs** — extend the `getAutoAttrs()` blocks in both `compileLessonHtml` and `buildPreviewElementHtml` if the type needs lesson-level attributes (e.g. `audio-listening` for `listening`)

## 13. Adding a new activity (in-app) — checklist

1. Create `components/Activities/<Name>.tsx`
2. Add the data shape to `types.ts` (e.g. `MyActivity` + extend `StandardLessonContent.activities`)
3. Add a `useState` slice and a score handler in `hooks/useWorksheetScores.ts` (if it's score-tracked)

## 14. Conventions

- **File-level**: all new code uses TypeScript; `.tsx` for components, `.ts` for utilities/services/hooks
- **Imports**: prefer named imports; group by external → internal → relative
- **Styling**: Tailwind only. The project uses Tailwind 4 (no `tailwind.config.js` — config is in `index.css` via `@theme` and `@import "tailwindcss"`)
- **Component naming**: PascalCase file + export, `export const Foo: React.FC<Props>` form
- **No comments unless asked** (per OpenCode global policy)
- **Errors**: surface as `setError(...)` in admin UI, or throw with a user-readable message in hooks
- **Server-side JSVM hooks**: keep them thin; put reusable logic in a sibling `*.js` and `require()` it
- **PocketBase SDK**: `pb.autoCancellation(false)` is set globally so React Query can issue concurrent reads safely. Do **not** flip it back.

## 15. Common pitfalls

- **`htmlCompiled` not in sync** — if you change `compileLessonHtml`, also update `compileHtmlForRecord` in `services/pocketbase.ts` so the server-side recompute matches. Or supply `htmlCompiled` from the editor (preferred — already done for all admin writes)
- **Custom-element double-registration** — both `LessonView` and `WebComponentPreview` guard script injection with a stable id; never append a script tag unconditionally
- **PocketBase JSON round-trip** — when sending `content: <HTML string>`, it goes through `JSON.stringify(string)` → `'"<html>"'`. PocketBase parses this and stores the string. On read, `lesson.content` is the original string. Always use `parseContent` to detect which case you're in
- **`<lesson-component>` placeholder** — if a user writes `<lesson-component />` (self-closing) in `html`, the regex in `compileLessonHtml` handles it; if they write it inline with text, the regex still matches
- **TinyMCE assets** — `public/tinymce/` is gitignored. The postinstall hook copies from `node_modules/tinymce`. If you bump the TinyMCE version, re-run `npm run postinstall` (or `node scripts/copy-tinymce.js`) before building
- **`base: './'`** in `vite.config.ts` — required because the app is served under a sub-path on the production server (`BASE_PATH=/worksheets`)
- **Genuine typos** — `parseContent` returns a helpful `error` field; if you see "Custom element tags must contain a hyphen" the user pasted a regular HTML tag like `<div>` or `<p>`. That's the most common editor mistake
